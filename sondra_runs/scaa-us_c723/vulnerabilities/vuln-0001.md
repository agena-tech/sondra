# MySQL Server Exposed to the Internet (TCP/3306) — scaa.us

**ID:** vuln-0001
**Severity:** HIGH
**Found:** 2026-04-27 18:13:45 UTC
**Target:** scaa.us (IP: 143.95.111.248) — TCP/3306
**Endpoint:** tcp/3306
**CVSS:** 8.3

## Description

A MySQL server (MySQL 5.7.23-23) was observed listening on TCP/3306 and reachable from the public internet at IP 143.95.111.248. Version/banner information was collected. Limited, non-destructive authentication attempts were performed using an explicit short list of credentials and were rejected (MySQL error 1045). Public exposure of database services increases the likelihood of automated scanning, credential guessing, exploitation of known MySQL vulnerabilities, and potential data exfiltration.

## Impact

An attacker who can authenticate to or exploit vulnerabilities in this publicly reachable MySQL instance could read, modify, or delete sensitive data; extract credentials or secrets stored in the database; disrupt availability; and use the host as a pivot to access internal systems. This materially increases regulatory, operational, and reputational risk for the organization.

Root cause
  The database daemon is reachable from the internet because TCP/3306 is permitted to the host by network/firewall rules and/or the MySQL instance is bound to an externally accessible interface. The server runs MySQL 5.7.23-23; older or unpatched 5.7.x releases are known to have published CVEs. Public reachability of the DB increases the attack surface for credential-guessing, automated scanners, and possible exploitation of known or unknown vulnerabilities even when credentials are not immediately weak.

Notes on validation
  - Only non-destructive checks were performed (service/version probe and a limited, explicit credential list). No destructive actions or data modification were attempted.
  - No successful authentication was achieved with the tested credentials; therefore, no database contents were read or exported during validation.

## Proof of Concept

Reproduction (non-destructive; for triage/forensics)
  1) Service/version probe (non-destructive)
     - Command:
       nmap -n -Pn -sV -p 3306 --script=mysql-info 143.95.111.248
     - Expected result: service identified as MySQL with version/banner information (as shown in the technical analysis).

  2) TCP banner grab (non-destructive)
     - Example command:
       python3 -c "import socket; s=socket.socket(); s.settimeout(5); s.connect(('143.95.111.248',3306)); print(s.recv(1024)); s.close()"
     - Expected result: initial handshake/banner bytes printed (may be non-human-readable protocol bytes).

  3) Limited, explicit authentication attempts (non-destructive)
     - Use a safe client (PyMySQL) and attempt a very small explicit set of credentials only:
         root:''    (empty)
         root:'root'
         root:'password'
         admin:'admin'
     - Expected result: authentication rejected with MySQL error 1045 (Access denied) if credentials are not valid.
     - Caution: Do NOT perform brute-force, credential spraying, or any destructive tests without explicit owner authorization.

  Evidence capture
    - Record timestamps and full command output for all steps for forensic review.
    - If any authentication succeeds, stop further activity and preserve a full, read-only transcript of queries executed; do not alter data.

```
#!/usr/bin/env python3
import socket
import sys
try:
    import pymysql
except Exception:
    pymysql = None

TARGET = "143.95.111.248"
PORT = 3306
TIMEOUT = 5

def grab_banner(target, port):
    s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    s.settimeout(TIMEOUT)
    try:
        s.connect((target, port))
        banner = s.recv(1024)
        print("BANNER:", banner)
    except Exception as e:
        print("BANNER ERROR:", e)
    finally:
        try:
            s.close()
        except Exception:
            pass

def try_auth(user, password):
    if not pymysql:
        print("PyMySQL not installed; skipping auth attempts")
        return
    try:
        conn = pymysql.connect(host=TARGET, user=user, password=password, port=PORT, connect_timeout=TIMEOUT)
        cur = conn.cursor()
        cur.execute("SHOW DATABASES;")
        dbs = cur.fetchall()
        print(f"AUTH SUCCESS: {user}@{TARGET}:{PORT} -> databases: {dbs}")
        conn.close()
    except pymysql.err.OperationalError as e:
        print(f"AUTH FAILED: {user}@{TARGET}:{PORT} -> {e}")
    except Exception as e:
        print(f"AUTH ERROR: {e}")

if __name__ == "__main__":
    print(f"Testing banner and limited auth against {TARGET}:{PORT}")
    grab_banner(TARGET, PORT)
    # Non-destructive explicit list only
    creds = [("root", ""), ("root", "root"), ("root", "password"), ("admin", "admin")]
    for u, p in creds:
        try_auth(u, p)
```

## Remediation

Immediate actions (highest priority)
  - Block or restrict TCP/3306 at the network perimeter. If external DB access is not required, block the port entirely in firewall/ACLs.
  - If remote database access is required, restrict it to a minimal allowlist of trusted management IPs or require access via a VPN/bastion host. Do NOT expose the DB directly to the internet.

MySQL configuration hardening
  - Bind MySQL to internal interfaces only (e.g., set bind-address to 127.0.0.1 or the internal management network) or enable skip-networking if appropriate.
  - Disable remote root login and remove any anonymous or unused accounts.
  - Enforce strong, unique passwords for all accounts and apply least-privilege for application credentials.
  - Enable TLS for MySQL connections (require_secure_transport=ON) to protect credentials in transit.
  - Enable and monitor authentication logs; alert on repeated failed logins and new source IPs.
  - Review user grants and remove unnecessary privileges.

Patch management & follow-up
  - Plan and execute an upgrade to a supported, patched MySQL release and review vendor advisories for CVEs affecting MySQL 5.7.23-23.
  - After remediation, retest from the internet to verify that TCP/3306 is not reachable and confirm that only approved management patterns (VPN/bastion) can reach the service.
  - If remote administrative access must remain, implement just-in-time access/temporary allowlisting with strong multi-factor authentication for the management channel.

Operational guidance
  - If any authentication succeeds during follow-up authorized testing, preserve a read-only transcript and halt further activity. Escalate immediately to incident response.
  - Consider moving database instances into private subnets with no direct internet egress and provide administrative access via hardened bastion hosts or jumpboxes.
