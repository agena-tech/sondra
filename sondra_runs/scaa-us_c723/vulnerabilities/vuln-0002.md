# SQL Injection in article.php and news.php id parameter — scaa.us

**ID:** vuln-0002
**Severity:** CRITICAL
**Found:** 2026-04-27 18:30:30 UTC
**Target:** http://scaa.us (143.95.111.248) — TCP/80; affected application paths: /article.php and /news.php (GET parameter id)
**Endpoint:** /article.php, /news.php
**Method:** GET
**CWE:** CWE-89
**CVSS:** 9.1

## Description

A SQL injection vulnerability was identified in the id query parameter of /article.php and /news.php on scaa.us (IP: 143.95.111.248). Supplying single-quote characters (') and encoded variants caused the application to return SQL error fingerprints and large response differences consistent with a server-side SQL parsing error. All testing was limited to non-destructive checks; no data extraction or destructive actions were performed.

## Impact

Successful exploitation of this SQL injection could allow an attacker to read or modify backend database contents. An attacker who is able to craft successful queries could exfiltrate sensitive data (user records, credentials, configuration), alter or delete data, and potentially pivot to other systems using retrieved credentials or configuration. This vulnerability therefore poses a significant confidentiality and integrity risk to the application and its users, with potential regulatory, reputational, and operational impact.

## Technical Analysis

Summary of evidence (verbatim excerpts taken from validation artifacts):

Request artifact (sqli_requests.txt)
2026-04-27T18:26:12Z - Request:
GET /article.php?id=1%27 HTTP/1.1
Host: scaa.us

Response artifact (sqli_responses.txt) — response excerpt:
... Warning: mysql_fetch_assoc() ...
... SQL syntax ...
... PDOException ...

Representative observation from artifacts:
- GET /article.php?id=1' produced SQL-related warning text and altered page content relative to the baseline request.
- GET /news.php?id=2%27 produced a large content-length delta and matched SQL error pattern signatures.

Detection methodology and rationale:
- Baseline request was captured and compared to responses containing injected meta-characters.
- Tested payloads: single-quote ("'"), encoded quote ("%27"), logical payloads ("' OR '1'='1", "' AND '1'='2").
- Detection oracles used: regex matching for common SQL error strings ("SQL syntax", "Warning: mysql_", "PDOException"), and response-length delta heuristic (payload vs baseline; evidence showed deltas >200 bytes).
- Multiple requests and payload variants produced consistent error fingerprints and length deltas across both /article.php and /news.php, providing a reliable oracle for injection detection.
- All checks were non-destructive and limited to detection only; no database extraction or destructive payloads were used.

Confidence: High — consistent error fingerprints across multiple payloads and endpoints plus substantial, reproducible response differences constitute strong evidence of injectable input handling in the id parameter.

2) Send a single-quote payload and capture response

```
#!/usr/bin/env python3
import re
import requests
import sys
from urllib.parse import quote_plus

TARGET = "http://scaa.us"
TIMEOUT = 10
SQL_ERROR_RE = re.compile(r"SQL syntax|Warning: mysql_|PDOException|mysql_fetch_assoc", re.IGNORECASE)

def fetch(path):
    url = TARGET + path
    try:
        r = requests.get(url, timeout=TIMEOUT, headers={"User-Agent":"Mozilla/5.0 (compatible)"})
        return r.status_code, r.text, len(r.content)
    except Exception as e:
        print(f"ERROR fetching {url}: {e}")
        return None, "", 0

def check_param(path, param, value):
    sep = '&' if '?' in path else '?'
    full = f"{path}{sep}{param}={quote_plus(value)}"
    status, body, length = fetch(full)
    return status, body, length

if __name__ == '__main__':
    # Baseline
    status_b, body_b, len_b = fetch("/article.php?id=1")
    # Single-quote payload
    status_p, body_p, len_p = check_param("/article.php", "id", "1'")
    print("Baseline length:", len_b, "Payload length:", len_p)
    if SQL_ERROR_RE.search(body_p):
        print("SQL error pattern detected in payload response")
    elif abs(len_p - len_b) > 200:
        print("Large content-length delta detected, possible injection oracle")
    else:
        print("No clear SQLi indicators detected")
```

## Remediation

Developer and operational remediation

Immediate (urgent)
- Remove public exposure: Block or restrict access to the application at the network perimeter if possible while a fix is applied (WAF rule to temporarily block requests containing raw single-quotes and common SQL meta-characters is acceptable as a short-term mitigation).
- Increase logging and monitoring for the affected endpoints and review historical logs for suspicious activity.

Application fixes (recommended)
- Enforce strong input validation and typing: treat id parameters as integers and reject non-numeric input at the application edge.
- Migrate all database access to use parameterized queries / prepared statements (no string concatenation of user input into SQL).
- Centralize database access and ensure the application uses least-privilege credentials (separate readonly/readwrite roles as appropriate).
- Harden database-facing controls: disable any database accounts that are not required; rotate credentials if compromise is suspected.

Operational and long-term
- Deploy a WAF tuned to block common SQLi payloads during the remediation window.
- Schedule an authorized, post-fix retest including authenticated and controlled enumeration only after fixes are deployed.
- Integrate automated scanning into CI/CD and add regression tests for injection/negative authorization cases.
- Conduct a code review focused on any other database-access paths that may perform manual string interpolation.
