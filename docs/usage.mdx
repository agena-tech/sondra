# Sondra Usage Guide

Sondra is a unified agent platform built for people who do not want to manage
a separate tool, script, or automation system for every task.

It brings general assistance, deep research, OSINT, penetration testing, ADB-based
phone integration, intelligent memory, and adaptive interaction into a single
cohesive environment.

This guide is the starting point for installation, configuration, scan modes,
providers, tools, integrations, automation, memory behavior, and development notes.

---

## What Sondra Does

Sondra is designed to provide a complete operational workspace for complex digital tasks.

Instead of forcing the user to switch between isolated agents, terminals, browsers,
proxies, scripts, and research tools, Sondra combines them under one intelligent
agent system.

Its purpose is simple:

> One platform. Multiple domains. Less fragmentation. More execution.

Sondra can assist with:

- General autonomous tasks
- Deep research and investigation
- OSINT workflows
- Penetration testing
- Web application testing
- Android device automation through ADB
- Local and cloud LLM workflows
- Long-term memory and contextual understanding

What makes Sondra different is not only what it can do, but how it adapts.

Sondra can transform emotional, contextual, and behavioral signals into structured
memory signals. This allows the system to understand the user more accurately over
time and respond with better context, tone, and continuity.

It is not only a task runner.

It is an adaptive agent platform.

---

## Core Capabilities

- Dynamic application testing with browser, proxy, terminal, Python, and notes.
- Multi-agent orchestration for complex targets and parallel investigation.
- OSINT, penetration testing, general assistance, and ADB phone integration.
- Interactive TUI for live work and non-interactive mode for automation.

## Quick Start

### Requirements

- Python 3.12+
- Docker running locally
- Poetry or pipx
- An LLM model configured through environment variables

---

### Install Poetry

```bash
curl -sSL https://install.python-poetry.org | python3 -
```

---

### Install Project Dependencies

```bash
poetry install
```

---

### Configure Your LLM

```bash
export SONDRA_LLM="openai/gpt-5"
export LLM_API_KEY="your-api-key"
```

---

### Audio Configuration

If you are running Sondra on WSL or a Windows environment, you need to manually
configure `ffplay` for audio playback.

Example:

```bash
export FFPLAY_COMMAND_DIR="/mnt/c/ffmpeg/bin/ffplay.exe"
```

Make it persistent on WSL:

To ensure this setting is applied automatically in every session, add it to your
`.bashrc` file:

```bash
echo 'export FFPLAY_COMMAND_DIR="/mnt/c/ffmpeg/bin/ffplay.exe"' >> ~/.bashrc
```

Then reload your configuration:

```bash
source ~/.bashrc
```

Linux systems:

On native Linux environments, installing FFmpeg is sufficient:

```bash
sudo apt install ffmpeg
```

Notes:

- `ffplay` is required for audio playback in Sondra.
- On Windows/WSL setups, make sure the path correctly points to your FFmpeg installation.
- You can verify the installation with:

```bash
ffplay -version
```

---

### Check the CLI

```bash
poetry run sondra -h
```

First runs may pull a Docker sandbox image. Outputs are saved under run and log
folders depending on mode and configuration.

## CLI Reference

Common options:

- `-v, --version`: Prints the installed Sondra version.

- `-m, --scan-mode`: Selects the operating mode.
  Supported values: `general`, `osint`, `adb`, `pentest`.

- `-l, --scan-level`: Selects the depth of execution.
  Supported values: `quick`, `standard`, `deep`.

- `-t, --target`: Defines the target.
  Can be a URL, domain, repository, local path, or IP address.

- `--instruction`: Provides inline task instructions.

- `--instruction-file`: Reads detailed instructions from a file.

- `-n, --non-interactive`: Runs without the interactive TUI.

- `--config`: Uses a custom JSON configuration file.

- `--subagents`: Forces subagent count where supported.

- `--retry`: Retries failed LLM requests before surfacing the final error.

- `--voice-speech`: Speaks assistant chat replies with local TTS.

- `-a, --auto`: Infers mode, level, target, and instruction from natural text.

## Additional Runtime Features

Sondra includes several runtime options that are useful in daily work.

- Auto mode can turn a natural language request into mode, level, target, and
  instruction arguments.
- Retry mode can reconnect failed LLM requests when `--retry` is provided.
- Subagents can be forced with `--subagents` where the selected mode supports
  parallel work.
- Non-interactive mode can run Sondra in CI, scripts, and headless terminals.
- Instruction files can carry larger scopes, credentials, rules, and task plans.
- Voice speech can read assistant chat replies aloud when `--voice-speech` is
  enabled.
- The interactive TUI can show live agent state, traffic, task status, memory
  layers, and emotion weights.

## Scan Modes

Use the mode that matches the job.

### `general`

General autonomous assistant mode.

This mode is intended for ordinary assistant behavior, planning, contextual help,
memory-aware responses, scheduled tasks, and long-term user continuity.

### `osint`

Open-source intelligence and research mode.

This mode is intended for public information gathering, entity research, exposed
asset discovery, reputation checks, and investigation workflows.

### `adb`

Android device automation mode.

This mode is intended for instruction-driven phone control through ADB connectivity.

### `pentest`

Penetration testing mode.

This mode keeps Strix-compatible security testing behavior while extending the
overall platform with Sondra-specific architecture, memory, and interaction layers.

---

## Scan Levels

### `quick`

Fast execution for smoke tests, CI checks, and urgent tasks.

### `standard`

Balanced execution for routine analysis and everyday workflows.

### `deep`

Thorough exploration for difficult targets, release reviews, audits, and complex
investigations.

## Custom Instructions

Instructions tell Sondra what to focus on, what to avoid, what credentials to use,
and what kind of result you expect.

Inline instructions are best for short tasks:

```bash
poetry run sondra -m osint -l standard --instruction "Focus on public mentions and leaked credentials"
```

Instruction files are better for larger scopes:

```bash
poetry run sondra -m pentest -l deep --target https://app.example.com --instruction-file ./scope.md
```

A good instruction file should include:

- Credentials and roles
- In-scope and out-of-scope areas
- High-priority endpoints or features
- API keys, headers, or test accounts
- Known risks, assumptions, and limitations
- Reporting expectations
- Rules of engagement

Clear instructions produce better results.

## Configuration

Sondra reads configuration from environment variables and optional config files.

Use `SONDRA_*` names for Sondra-specific settings.

Important variables:

- `SONDRA_LLM`: Required model name in LiteLLM format.
  Example: `openai/gpt-5`.

- `LLM_API_KEY`: Provider API key when needed.

- `LLM_API_BASE`: Custom endpoint for local or compatible APIs.

- `OPENAI_API_BASE`, `LITELLM_BASE_URL`, `OLLAMA_API_BASE`:
  Alternative base URLs.

- `LLM_TIMEOUT`: LLM request timeout.

- `SONDRA_REASONING_EFFORT`: Reasoning effort for supported reasoning models.

- `PERPLEXITY_API_KEY`: Enables web search and research tools.

- `SONDRA_MEMORY_DIR`: Memory database directory.

- `SONDRA_MEMORY_LOG_DIR`: Memory log directory.

- `SONDRA_MEMORY_EMBED_MODEL`: Embedding model for memory search.

- `SONDRA_MEMORY_EMBED_BASE_URL`: Embedding provider endpoint.

- `SONDRA_MEMORY_SYNC_MODE`: Memory sync behavior.

- `SONDRA_MEMORY_TEMPORAL_DECAY`: Enables memory importance decay.

- `SONDRA_IMAGE`: Docker sandbox image.

- `SONDRA_RUNTIME_BACKEND`: Runtime backend, normally `docker`.

- `SONDRA_SANDBOX_EXECUTION_TIMEOUT`: Tool execution timeout inside the sandbox.

- `SONDRA_SANDBOX_CONNECT_TIMEOUT`: Tool server connection timeout.

- `SONDRA_DISABLE_BROWSER`: Disables browser tools when set to `true`.

- `SONDRA_ADB_URL`: HTTP bridge URL for ADB mode.

- `SONDRA_ADB_TIMEOUT`: ADB bridge request timeout.

- `SONDRA_ADB_SCREENSHOT_PATH`: Path used for binary ADB screenshots.

## Memory System

Sondra includes an intelligent memory system developed by Agena Memory Systems.

Memory is primarily used in general mode to help the assistant retain useful
context across turns and future sessions.

It can store:

- Conversation history
- Semantic facts
- User profile data
- Active task states
- Emotional signals
- Scheduled tasks
- Long-running context

The goal is not to make the model permanently trained on the user.

The goal is to provide the right context at the right moment.

> Memory is not model training.
> Memory is structured context.

---

## How Learning Works

Sondra can persist useful information from interaction history and organize it
into structured memory.

Learning may include:

- Saving user and assistant turns as conversation messages.
- Extracting important user facts into semantic memory.
- Storing stable personal facts as profile facts.
- Saving recent task progress as task state.
- Tracking emotional signals when they are relevant to response quality.
- Storing scheduled tasks separately for dispatch.

This allows Sondra to keep continuity without forcing the user to repeat the same
information again and again.

---

## How Recall Works

Before generating a response, Sondra can build an automatic memory context.

This context may combine:

- Recent conversation lines
- Exact matches
- Keyword search
- Semantic search
- Profile facts
- Active task state
- Emotion-aware signals

Hybrid retrieval can use both lexical search and embeddings.

MMR ranking helps reduce repetition and noise.

Temporal decay can reduce the importance of older memories over time, allowing
newer and more relevant information to become stronger in context.

---

## Important Memory Files and Folders

- `SONDRA_MEMORY_DIR`: Where `memory.db` is stored.
- `SONDRA_MEMORY_LOG_DIR`: Where memory logs and events are written.
- `memory_events.jsonl`: Structured memory event log.
- `memory_debug.log`: Debug-level memory activity log.
- `config/memory_config.yaml`: Default memory configuration.

---

## Memory Behavior Notes

- Scheduled tasks are intended for `general` mode only.
- Non-general modes such as `osint`, `adb`, and `pentest` do not use the task scheduling panel.
- Memory retrieval is prompt context, not model fine-tuning.
- Embedding-backed semantic search works best when the local embedding backend is reachable.
- Memory quality depends on clean extraction, good ranking, and useful context boundaries.
- Memory marker and signal detection is designed for Turkish and English only.
- Other languages can still be stored as raw conversation history, but semantic
  markers, task markers, and emotion markers are only expected to work reliably
  in Turkish and English.

## Task Creation And Scheduling

Scheduled tasks are a general-mode memory feature.

When the user writes a time-based request, Sondra can detect the schedule,
create a task record, store it in memory, and later dispatch it through a
dedicated task agent.

Example requests:

```text
saat 20:40 ta google.com'a gir
20 saniye sonra google.com'a gir
her 20 saniyede bir google.com'a gir
yarin saat 09:00'da raporu kontrol et
```

The task flow is:

1. The user writes a time-based instruction.
2. Sondra extracts the task text and schedule.
3. A task row is saved to the scheduled task store.
4. The TUI displays the task in the `SUBSYSTEM MENU`.
5. When the time arrives, Sondra creates a task subagent.
6. The task subagent executes the task and reports back to the root agent.
7. The root agent summarizes the result for the user.

Task labels in the panel:

- `20:40`: A clock-based scheduled task.
- `00:20s`: A task scheduled in seconds.
- `20:00m`: A task scheduled in minutes.
- `+`: A recurring task.
- `EXEC`: A task is currently executing.
- `CMPLT`: A task recently completed.
- `--:--`: Empty slot or inactive task entry.

Task scheduling is not used by `osint`, `adb`, or `pentest` mode. In those
modes the task panel shows that scheduled tasks are supported only in general mode.

## Emotion Signals

Sondra can score emotional signals from user messages so replies can adapt tone
and urgency. This is not a diagnosis and not a permanent personality profile.
It is a lightweight interaction signal used to make the assistant respond more
appropriately.

Tracked emotion categories include:

- Happiness or positive tone.
- Sadness or low mood.
- Stress, frustration, or pressure.
- Anger or hostile wording.
- Neutral/default tone.
- Confidence score for how strong the detected signal is.

How emotion increases:

- Stronger words increase the matching emotion score.
- Repeated emotional wording can make the current signal stronger.
- Intensifiers such as "very", "really", "cok", or similar phrases can raise
  intensity.
- Direct phrases such as feeling bad, being frustrated, being happy, or being
  angry can push the relevant category upward.

How emotion weakens:

- Later messages without the same signal reduce the live emotional weight.
- Neutral or calmer messages move the current state back toward neutral.
- Softer wording produces lower confidence.
- The emotion panel compares recent/current scores, so arrows can go down when
  the latest signal is weaker than the previous one.

How Sondra uses it:

- If sadness or stress is detected, replies can become calmer and more supportive.
- If frustration is detected, replies can become more direct and practical.
- If neutral is dominant, replies stay normal.
- The model receives emotion context as prompt context; it is not retrained.

---

## Learning From Corrections And Mistakes

Sondra can learn from interaction signals, but it does so through memory updates,
not by changing the model weights.

When the user corrects the assistant, says something was wrong, or gives negative
feedback, the system can treat that as a memory feedback signal. Useful memories
can be reinforced, noisy memories can lose importance, and future retrieval can
prefer more reliable context.

Examples of useful feedback:

```text
hayir, boyle yapma
bu yanlis oldu
bir dahaki sefere daha kisa cevap ver
bunu hatirla
evet, bu dogru
```

What can happen internally:

- The correction is saved in conversation history.
- Relevant semantic memory can be updated or reinforced.
- Bad or noisy memory can be given lower importance.
- Future prompts can include the corrected preference or fact.
- Repeated successful recall can increase importance slightly.
- Old or unused semantic memory can decay over time.

This means Sondra can improve continuity and behavior across turns, but the
underlying LLM is still the same model. Learning happens through stored context,
ranking, feedback, and retrieval.

## Interactive TUI Screen

The TUI is the live control screen shown during interactive runs.

Main areas:

- `TERMINAL`: The large left area. It shows boot output, user messages,
  assistant replies, tool activity, task messages, waiting state, and final
  results.
- `AGENT MAP`: The upper-right tree. It shows the root agent and any subagents
  currently created for the task.
- `SUBSYSTEM MENU`: The middle-right compact panel. It shows operational pages
  such as graph, agent info, resources, task queue, memory layers, and emotion.
- Stats area: Shows request/response rates, sent/received traffic, model, cost,
  Caido port, and version/status details.
- Bottom input: Where the user sends messages while the root agent is waiting.

Useful controls:

- `Ctrl + Q`: Request exit.
- `Esc`: Stop/interruption behavior where supported.
- `Up/Down`: Scroll terminal history.
- Left/right arrows in `SUBSYSTEM MENU`: Switch the subsystem page.
- Center button in `SUBSYSTEM MENU`: Stop or resume selected agent depending
  on current state.

The TUI is designed to make the system observable: you can see what the agent is
doing, whether subagents exist, what mode is running, whether memory is active,
and whether scheduled tasks are waiting or executing.

## Voice Speech

Use `--voice-speech` to enable local speech output for assistant chat replies.

```bash
poetry run sondra -m general --voice-speech
```

What it speaks:

- Assistant chat replies.
- Natural responses after memory and task events.

What it does not speak:

- Raw tool output.
- Raw command output.
- Internal tool blocks.
- Empty control messages.

How playback works:

- Speech uses local Piper voice models.
- Turkish is the default voice path.
- English is used only when English is detected and an English model is available.
- On Linux, playback uses `ffplay` from `PATH`.
- On WSL or Windows-style setups, set `FFPLAY_COMMAND_DIR` to the ffmpeg binary
  directory.

Example WSL setup:

```bash
export FFPLAY_COMMAND_DIR="/mnt/c/ffmpeg/bin"
```

Controls:

- `+`: Increase voice volume in the TUI.
- `-`: Decrease voice volume in the TUI.

Language support:

- Speech supports Turkish and English only.
- Other languages fall back to Turkish-style speech handling.

---

## Language Support

Sondra's language-aware memory and speech features are intentionally limited to
Turkish and English.

- Memory signal support: Turkish and English.
- Speech/TTS support: Turkish and English.
- Marker-based learning, emotion detection, task phrasing, and speech language
  switching are not guaranteed outside Turkish and English.
- Non-TR/EN text may still be saved in conversation history, but advanced signal
  extraction should be treated as best effort only.

## SUBSYSTEM MENU Panel

The `SUBSYSTEM MENU` is a compact multi-page status panel.

Use the `<` and `>` arrow buttons to move between pages.

Pages:

### Page 1: Activity Graph

Shows compact request/response activity over time.

Use it to see whether the agent is actively sending requests, receiving replies,
or staying idle.

### Page 2: Agent Info

Shows the current model, mode, live agent count, progress state, and elapsed time.

Common fields:

- `LLM`: Current model family.
- `MODE`: Current mode such as `GENERAL`, `OSINT`, or `ADB`.
- `AGNTS`: Number of visible agents.
- `PROGR`: Running or stopped state.
- `TIME`: Elapsed runtime.

### Page 3: System Resources

Shows CPU, RAM, disk, swap, and free memory indicators.

This helps identify whether slow behavior may be caused by system pressure.

### Page 4: Task Queue

Shows scheduled tasks in general mode.

This is where entries like `20:40`, `00:20s`, `20:00m`, `EXEC`, `CMPLT`, and `+`
appear.

If the current mode is not `general`, scheduled tasks are not used and the panel
shows that general mode is required.

### Page 5: Memory Layers

Shows the current memory signal state.

Common fields:

- `LAYERS`: Number of active memory-related layers.
- `SIGNAL`: Latest memory signal type.
- `ENTRY`: Latest memory entry code.
- `WEIGHT`: Importance value.
- `CONFIDENCE`: Confidence value.
- `REASON`: Whether the memory signal is active.

This page helps confirm whether memory writes and memory updates are happening.

### Page 6: Emotion Weights

Shows current emotion weights.

Common fields:

- Happiness/positive tone.
- Sadness.
- Stress/frustration.
- Neutral.
- Confidence/trend arrows.

Up arrows mean the latest signal is stronger than before. Down arrows mean it is
weaker than before. Stable values mean no major change was detected.

## CI/CD Integration

Use non-interactive mode for automation:

```bash
poetry run sondra -n -m pentest -l quick --target ./
```

Typical exit behavior:

- `0`: Completed without vulnerabilities.
- `1`: Execution error.
- `2`: Vulnerabilities found in headless mode.

GitHub Actions, GitLab CI, Jenkins, and CircleCI workflows can follow the same
basic structure:

1. Install Sondra.
2. Configure LLM secrets.
3. Run quick scans on pull requests.
4. Run deeper scans on schedules or release candidates.
5. Store reports as build artifacts.
